WORTH 6. Stacked retry logic
transport retries × application retries = 15x API calls · toggle evidence
# Layer 1: Transport retry in aiService.js async function callWithRetry(fn, retries = 2) { ... } ← Retries on network errors, 429, 5xx # Layer 2: Generation service retry in generation/*.js async function generateWithRetries(...args) { const MAX_RETRIES = 5 ← Retries on quality/parse failures ... } # Combined worst case: aiService.callWithRetry(2) × generateWithRetries(5) = 15 API calls ↑ transport retries ↑ application retries # No backoff coordination between layers: First call fails (network blip) → transport retry (2) → succeeds → generation parse fails → app retry (5) Each app retry also has transport retries If both trigger simultaneously, total = 15 calls

Retries are stacked at two levels: aiService.js retries at the transport level (network errors, HTTP 429/5xx) with 2 retries, while individual generation services (generation/*.js) retry at the application level (parse failures, quality gates) with 5 retries. In the worst case — a network blip during the last application retry — the system could make 15 API calls. The architecture needs a unified retry policy: total retries across all layers should be bounded.
