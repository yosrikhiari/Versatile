Idempotency Strategies

StrategyMechanismStorageBest ForDownside
Claim Check INSERT ... ON CONFLICT or SET NX to claim a key; return cached result on repeat SQL table or Redis Payment, credit deduction, any POST that produces a side effect Extra write per request; TTL management
Optimistic Lock Include a version/timestamp in the request; reject if stale Row version column PUT/PATCH on a resource the client has already read Client must track version; first writer wins
Idempotent Key in URL Client generates a UUID and appends it to the URL; server deduplicates Cache or DB with TTL External integrations, webhook callbacks Key collision risk; key management burden on client
Natural Idempotency The operation is inherently repeatable (e.g., SET current_balance = 100) None needed State-setting operations, upserts Rare; most mutations are not naturally idempotent

Key Design Decisions

QuestionOption AOption B
Who generates the key? Server — returns Idempotency-Key in response header; client uses it on retry Client — sends Idempotency-Key header; simpler for external callers
Storage backend SQL — transactions are atomic; natural fit with existing data Redis — faster reads/writes; built-in TTL; but one more service to manage
TTL duration 24 hours — Stripe standard; covers most retry windows Configurable — longer for async workflows, shorter for latency-sensitive APIs
Response on repeat 200 OK — return cached response; transparent to caller 409 Conflict — reject repeat with different payload; safer for non-idempotent clients

Failure Modes

  • Key collision — client generates the same key for different requests. Mitigation: include a client-ID prefix in the key namespace.
  • TTL expiry mid-retry — client retries after the key expired; server processes the request again. Mitigation: set TTL longer than the maximum expected retry window.
  • Partial execution — server crashes after claiming the key but before persisting the result. Mitigation: use a two-phase claim (claim → execute → complete) with a recovery scan.
  • Clock skew — distributed key generation relies on timestamps that differ across servers. Mitigation: use UUIDs, not timestamp-based keys.
