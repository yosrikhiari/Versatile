Idempotency Reference

Key concepts, patterns, and gotchas for safe retry APIs.

Lesson 3 · Quick reference

Definition

An operation is idempotent when performing it multiple times produces the same side-effects as performing it once. Not the same response — the same resulting state.

Idempotency Key Pattern

Client Contract

  • Generate one key per logical operation (UUIDv4, generated when user confirms the action).
  • Send it as Idempotency-Key header on every POST / PATCH.
  • Reuse the same key across all retries of that operation — creating a new key per HTTP attempt defeats the mechanism.
  • Retry on network errors and 5xx; never retry 4xx.
  • Use exponential backoff + jitter to avoid thundering herd.

Server State Machine

1Claim — Atomically insert a row (key, status=in_progress). Use INSERT ... ON CONFLICT DO NOTHING — exactly one concurrent request wins.
2Check — If row exists with status=done, compare stored request hash. Match → replay cached response. Mismatch → 422 Unprocessable.
3Execute — Run the real handler, saving response code + body to the key row. Mark status=done.
4Reap — Purge keys older than TTL (24h is standard). Emit Idempotent-Replayed: true header on cached responses.

Storage Schema

idempotency_keys (
  id           UUID PRIMARY KEY,
  scope        TEXT NOT NULL,          -- e.g. "POST /api/charges"
  scope_id     TEXT NOT NULL,          -- e.g. user/account ID
  key          TEXT NOT NULL,          -- client-provided Idempotency-Key
  request_hash TEXT NOT NULL,          -- SHA-256 of method + path + body
  status       TEXT NOT NULL DEFAULT 'in_progress',  -- in_progress | done
  response_code   INTEGER,
  response_body   JSONB,
  recovery_point  TEXT,                -- for multi-step workflows
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  locked_at    TIMESTAMPTZ,
  UNIQUE (scope, scope_id, key)
)

Pitfalls

PitfallFix
Check-then-insert raceUse INSERT ... ON CONFLICT — one atomic statement
Global (cross-tenant) keysScope by (endpoint, account_id)
No request hashStore hash; reject mismatches with 422
Side effects outside transactionEnqueue transactionally; deduplicate in worker
Key table grows foreverTTL purge job (24h), index on created_at
Caching transient errorsOnly cache 4xx / terminal; let 5xx retry naturally
New key per HTTP attemptGenerate once at UX gesture; reuse on every retry

Signals

200 + Idempotent-Replayed: true
Safe to ignore — means "this is a retry, nothing new happened"
409 Conflict
Another request with this key is currently in-progress. Retry after a short delay.
422 Unprocessable
Same key used for a different request body — client bug.
Missing Idempotency-Key
Server rejects critical POSTs that lack the header. This is a feature, not a bug.
