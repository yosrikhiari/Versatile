Quick-reference glossary for the architectural patterns in Versatile.

Offline-First

Data is written to IndexedDB (via Dexie.js) before it is synced to the backend. The app is fully functional without a server. The backend is a replica, not the source of truth.

Key Filessrc/services/db-core.js (schema), src/services/sync-engine.js (sync)
Status TrackingsyncStatus column per table: pending-create | pending-update | pending-delete | synced | synced-deleted
Sync Interval30 seconds (configurable)
Conflict StrategyLast-write-wins
StrengthsZero-infrastructure startup, works offline, backend is optional
Trade-offsConflict resolution complexity, potential sync delays, API keys in browser

Client-Side AI

The browser calls AI providers (OpenAI, Anthropic, Gemini, Groq, Ollama) directly. The server never handles AI traffic — it's not a bottleneck for the core feature.

Entry Pointsrc/services/aiService.js
Provider Implementationssrc/services/providers/{openai,anthropic,gemini,groq,ollama}.js
Key FeatureProvider abstraction: all providers export the same 4 functions
ConcurrencySemaphore per provider (1 for Ollama, 4 for cloud providers)
StrengthsServer never bottlenecks on AI, easy to add providers, parallel generation
Trade-offsAPI keys in browser storage (encrypted), no server-side usage tracking

Clean Architecture (CQRS)

Backend organized in 4 concentric layers. Dependencies point inward — Domain knows nothing, API knows everything.

ProjectResponsibilityDependencies
Versatile.Domain/Entities, enums, interfacesNone
Versatile.Application/CQRS commands/queries, handlers, DTOs, validators, AutoMapper profilesDomain
Versatile.Infrastructure/EF Core DbContext, repositories, middleware, SignalRApplication + Domain
Versatile.Api/Controllers, Hubs, Program.cs, JWT auth configAll three

CQRS Triad

Each CQRS feature has three files:

FileJobExample
*Command.cs / *Query.csInput contract — what does this operation need?CreateStoryCommand.cs
*Handler.csBusiness logic — what does this operation do?CreateStoryHandler.cs
*Validator.csGuard — is the input valid?CreateStoryValidator.cs

Modular Frontend (Features + Composables)

Frontend code is grouped by feature, not by type. Each feature directory owns its components. Reusable stateful logic lives in composables.

LayerLocationJob
Feature Componentssrc/components/{feature}/UI rendering for a feature
Composablessrc/composables/use{*}.jsReusable stateful logic (65+)
Pinia Storessrc/stores/{*}.jsGlobal state management (15 stores)
DB Servicessrc/services/db-{*}.jsIndexedDB persistence layer
Shared Primitivessrc/components/shared/Reusable UI (Modal, ErrorBoundary, BaseButton)

SignalR Runway

Real-time WebSocket hubs are deployed and JWT-authenticated but have no frontend client yet. Architectural runway for future features.

HubPurposeEndpoint
CollaborationHubMulti-user cursor sync, real-time content changes/hubs/collaboration
GenerationHubServer-side AI streaming (for when client-side isn't enough)/hubs/generation

Glossary

TermDefinition
IndexedDBBrowser-based NoSQL database. Dexie.js is a wrapper that makes it usable with promises.
Dexie.jsA minimal wrapper around IndexedDB that provides async CRUD, live queries, and change tracking.
PiniaVue 3's official state management library. Each store is a singleton holding reactive state for one domain.
ComposableA Vue 3 function that uses the Composition API (ref, computed) to encapsulate reusable logic.
CQRSCommand Query Responsibility Segregation — separate models for writes (commands) and reads (queries).
SignalRASP.NET Core's real-time WebSocket library. Provides persistent connections with group-based messaging.
AutoMapper.NET library for mapping between entity models and DTOs automatically.
SemaphoreConcurrency control primitive — limits how many operations can run simultaneously.
Architectural RunwayInfrastructure built before it's needed, to enable future development velocity.
