Multi-Tenancy Fundamentals

Why sharing one app between customers is harder than it looks.

Lesson 4 · 25 min · MISSION.md
Prerequisites: Lesson 1 (Five Patterns of Scalability) is recommended. You should be comfortable opening files in the backend/Versatile.Domain/, backend/Versatile.Infrastructure/, and src/stores/ directories.

Right now Versatile is a single-tenant app: one user, one database, one cookie jar. All the data belongs to you. There's no concept of an "organization" or "workspace" — just your projects, your stories, your settings.

Multi-tenancy is what happens when you want one running instance of the app to serve many customers (tenants), with each tenant unable to see anyone else's data. Think Google Drive, Notion, GitHub — you log in and see your stuff, not the whole world's.

This sounds simple. It is not. Every layer of the stack needs to be aware of "who" the current tenant is — and if even one layer forgets, data leaks.

1. The Three Isolation Models

There are three approaches to separating tenant data. They differ in cost, complexity, and how hard they are to recover from if something goes wrong.

ModelHow it worksSecurityCostOperational complexity
Isolated Database Each tenant gets their own PostgreSQL database. App connects to the right one at login. 🛡️ Best — data is physically separated 💰 Highest — N databases, N connection pools 🧠 High — migrations must run N times, connection routing logic needed
Shared DB, Schemas Per Tenant One database, but each tenant has their own schema (PostgreSQL schemas like tenant_42.tables). 🛡️ Strong — schema boundary prevents cross-tenant queries 💰 Medium — one database server, N schemas 🧠 Medium — SET search_path per connection, migration per schema
Shared DB, Shared Schema One database, one set of tables. Every row has a tenant_id column. Queries filter by it. (Versatile's future path.) 🛡️ Weakest — a missing WHERE tenant_id = @current leaks data 💰 Cheapest — one database, one schema, one connection pool 🧠 Highest discipline — SQL-level isolation is the hardest to enforce
Decision

Why Shared Schema for Versatile?

Versatile should use shared DB + shared schema. Here's why:

  1. 36 entity types — that's a lot of schemas to create per tenant for the schema-per-tenant model. Each migration runs N times.
  2. Collaboration — SignalR hubs need to work across tenants. Shared schema with RLS keeps the messaging layer simpler.
  3. Cost at launch — isolated databases for every customer start-up would multiply database costs before revenue justifies it.
  4. RLS is good enough — PostgreSQL Row-Level Security (RLS) USING (tenant_id = current_setting('app.tenant_id')) is battle-tested by hundreds of SaaS products.

The trade-off: shared schema requires discipline at every query site. One missing filter = one data leak. RLS is your safety net, but RLS itself needs to be turned on per table. This is why Phase 2 (middleware) is critical — it automates what would otherwise require human vigilance.

2. The 7 Gaps: How Versatile Fails Each One

Let's walk through the codebase and see each gap in the flesh.

Gap 1: No Organization Entity

Open backend/Versatile.Domain/Entities/BaseEntity.cs:

public abstract class BaseEntity
{
    public Guid Id { get; set; }
    public DateTime CreatedAt { get; set; }
    public DateTime UpdatedAt { get; set; }
}

public abstract class UserOwnedEntity : BaseEntity
{
    public string UserId { get; set; } = string.Empty;
}

There it is. UserOwnedEntity — the base class for all 36 content entities. It says "this data belongs to a user." There is no parallel OrganizationOwnedEntity. There is no Organization class anywhere in the Domain project.

Open backend/Versatile.Domain/Entities/Story.cs:

public class Story : UserOwnedEntity
{
    public string Title { get; set; }
    public string? Description { get; set; }
    // ...
}

A Story belongs to a user. In multi-tenant Versatile, a Story should belong to an Organization — and the user should belong to that Organization through a membership (which carries a role: owner, editor, viewer).

Gap 2: All Queries Filter by UserId, Not TenantId

Open backend/Versatile.Infrastructure/Data/ApplicationDbContext.cs and search for SaveChangesAsync or UserId. You'll find logic like:

// Simplified: sets UserId on entities before save
foreach (var entry in ChangeTracker.Entries<UserOwnedEntity>())
{
    if (entry.State == EntityState.Added)
        entry.Entity.UserId = _currentUserService.UserId;
}

This is the automatic ownership assignment — it happens at the DbContext level, before any entity hits the database. For multi-tenancy, you need a parallel mechanism: TenantId set automatically, and all queries filtered by it.

Gap 3: JWT Has No Org Claims

Open backend/Versatile.Infrastructure/Services/AuthService.cs. Find where the token is constructed:

var claims = new[]
{
    new Claim(ClaimTypes.NameIdentifier, user.Id),
    new Claim(ClaimTypes.Email, user.Email),
    new Claim(ClaimTypes.Name, user.DisplayName ?? user.Email),
    // ...
};

There's no org_id or org_role claim. After multi-tenancy, a user might belong to 3 orgs, have role "editor" in two and "viewer" in one. The JWT needs to carry the active session's org context: which org they're acting as right now, and what permissions that gives them.

Gap 4: No Tenant Middleware

Open backend/Versatile.Api/Program.cs. Look at the middleware pipeline (app.Use...() calls). There's no UseTenantResolution() — nothing that reads the X-Tenant-Id header (or extracts it from the JWT) and stores it in a scoped service that every query can reference.

In a multi-tenant system, the tenant resolution happens once per request, at the middleware layer. Every downstream query, every SignalR hub call, every background job — they all need access to CurrentTenant without having to re-derive it themselves.

Gap 5: SignalR Groups Not Namespaced

Open backend/Versatile.Api/Hubs/CollaborationHub.cs:

await Groups.AddToGroupAsync(Context.ConnectionId, $"collab_{storyId}");

In a single-tenant app, story IDs are globally unique. In a multi-tenant app, two tenants could both have a story with ID 42. Without tenantId_storyId as the group name, tenants would receive each other's real-time collaboration events.

Gap 6: AI Generation Blocks the Hub

Open backend/Versatile.Api/Hubs/GenerationHub.cs:

await foreach (var chunk in _generationService.GenerateStreamAsync(...))
{
    await Clients.Caller.SendAsync("generationChunk", chunk);
}

The AI generation runs inline in the SignalR hub handler. SignalR hub instances are scoped to a connection — if 10 users are generating AI content simultaneously on different connections, they each occupy a thread. This is fine for a single user. For multi-tenant SaaS, this blocks horizontal scaling: you can't move a generation task to a background worker because it's tied to the WebSocket connection.

Horizontal Scaling is the Hardest Gap

Gaps 6 and 7 are about infrastructure, not data. Moving AI generation off the hub requires a distributed job queue (RabbitMQ, Redis Streams, Azure Service Bus). Scaling SignalR across multiple servers requires a backplane (Redis, Azure SignalR Service). These are the last-phase changes because they depend on everything else being stable first.

Gap 7: No Distributed Cache or Queue

Search for IDistributedCache or Redis in backend/Versatile.Api/Program.cs — you won't find them. There's no distributed cache for session data, rate-limiting counters, or tenant metadata. There's no background job processor for offloading long-running work.

In a single-user app, in-memory everything is fine. In a multi-tenant app with N concurrent users, the app needs to share state across instances — which means a cache that's reachable by all instances, not just one.

3. Deep Dive: The Missing Organization

Let's model what the Organization entity looks like and trace the ripple effect through the codebase.

What we need

// New entity
public class Organization : BaseEntity
{
    public string Name { get; set; } = string.Empty;
    public string? Slug { get; set; }  // URL-friendly identifier
    public SubscriptionTier Tier { get; set; }
}

// New join entity
public class OrganizationMembership : BaseEntity
{
    public Guid OrganizationId { get; set; }
    public Organization Organization { get; set; }
    public string UserId { get; set; }
    public OrganizationRole Role { get; set; }  // Owner, Admin, Editor, Viewer
}

// Modified entity base
public abstract class OrganizationOwnedEntity : BaseEntity
{
    public Guid OrganizationId { get; set; }
}

// Membership methods
public interface IOrganizationService
{
    Task<Organization> CreateAsync(string name, string ownerUserId);
    Task AddMemberAsync(Guid orgId, string userId, OrganizationRole role);
    Task<IReadOnlyList<Organization>> GetUserOrganizations(string userId);
}

The ripple effect

This one new entity touches every layer:

LayerWhat changes
DomainAdd Organization, OrganizationMembership, OrganizationOwnedEntity base class. Add OrganizationRole enum.
ApplicationNew CQRS handlers: CreateOrganizationCommand, InviteMemberCommand, SwitchOrganizationCommand. Existing Story handlers now need org context.
InfrastructureNew OrganizationConfiguration (EF config), migration for tables. New OrganizationService. DbContext interceptor sets TenantId on saves.
APINew OrganizationsController. JWT now emits org_id, org_role. Tenant resolution middleware.
FrontendNew organizationStore in Pinia. Auth flow now includes org selection. API calls include X-Tenant-Id header. Sidebar shows org name + switch menu.

Now ask yourself: if you added an OrganizationId to the Story entity, which of these files would need to change? The answer reveals the scope of Phase 1.

Why Phase 1 starts here

The Organization entity is the foundation. Without it:

  • You can't add a tenant_id column to tables (what would it reference?)
  • You can't put org claims in the JWT (there's no org to claim)
  • You can't scope SignalR groups by tenant (no tenant ID to prepend)
  • You can't build a workspace switcher UI (nothing to switch between)

This is why Phase 1 in the roadmap is "Data Model & Auth": build the org, wire it into authentication, and lay the foundation for every subsequent phase.

Putting It Together

PhaseWhat it changesWhy that order
1. Data Model & Auth Organization entity, membership, JWT org claims Foundation — nothing else can work without the tenant concept existing
2. Middleware & Tenant Context Tenant resolution middleware, RLS on all tables, scoped CurrentTenantService Enforcement — the org exists, now make sure every query uses it
3. Scalability Infrastructure Redis cache, background job queue, SignalR backplane, async AI generation Scale — once data is isolated, make the app handle N tenants
4. Frontend Multi-Tenant UX Workspace switcher, org-scoped dashboards, invitation flows Experience — the user-facing part, last because everything else must be stable first
