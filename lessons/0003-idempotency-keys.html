Idempotency Keys

Making your API safe to retry — the most valuable pattern you'll learn.

Lesson 3 · 15 min · No prerequisites · Reference
Why this matters: Every API you build will eventually need to survive network failures, client crashes, and duplicate requests without creating chaos. Stripe processes billions of dollars per year and idempotency is the mechanism that makes it safe to retry. This lesson teaches you the pattern so you can apply it to any POST endpoint — including Versatile's own payment flows.

What Is Idempotency?

An operation is idempotent when performing it N times produces the same side-effects as performing it once.

It does not mean "returns the same response." It means the resulting state is the same no matter how many times you try.

Common Mistake

Same Response ≠ Idempotent

If you POST to create a charge and it returns { "id": "ch_123", "status": "succeeded" } on the first call and { "id": "ch_123", "status": "succeeded" } with the same id on the retry — that's idempotent. If the retry creates a second charge (ch_456) because you didn't check for the duplicate — that's not idempotent. The response looks the same but the state is different.

Think about it

Open src/services/aiService.js. Look at aiGenerate. This function sends a POST request to the AI provider. If the network fails mid-flight and you retry, do you get:

  1. Two different completions (new request, new response)?
  2. The same completion? (Only if the provider supports idempotency keys)

Most AI providers don't guarantee idempotency for chat completions. That means every retry risks spending money on a duplicate. This is exactly the problem the pattern solves.

HTTP Methods: The Built-In Idempotency

Before you reach for idempotency keys, know what HTTP gives you for free:

MethodSemanticsIdempotent?Why?
GET Retrieve a resource Yes Reading doesn't change state
PUT Replace a resource Yes PUT with the same body produces the same state
DELETE Remove a resource Yes Deleting a deleted thing is a no-op
POST Create or trigger No POST is the "do something" verb — it could be anything
PATCH Partial update No* Depends on the implementation; not guaranteed by spec
Key Insight

Use the Right Verb First

If you're designing an API and the operation can be modeled as "replace resource X" (PUT) or "remove resource X" (DELETE), use the built-in idempotency of those methods before reaching for keys. Only POST (and sometimes PATCH) need explicit idempotency support.

The Pattern: Idempotency Key Header

The standard pattern, proven at Stripe scale:

Client Side

  1. Generate a key — a UUID v4, created once when the user confirms the action (not once per HTTP attempt).
  2. Send it as a header — Idempotency-Key: uuid-v4-value on every POST/PATCH.
  3. Reuse on retry — if the first attempt fails with a network error or 5xx, send the same key again.
  4. Never reuse across different operations — each logical operation gets its own key. Buying a coffee and buying a sandwich are different operations, different keys.

Server Side

The server maintains a key-value store of previously-seen keys:

1Claim. Atomically insert (key, status=in_progress). Use INSERT ... ON CONFLICT DO NOTHING. Exactly one concurrent request wins; the rest see the existing row.
2Check. If row exists with status=done, compare a hash of the request body. Match → replay cached response. Mismatch → 422 Unprocessable Entity.
3Execute. Run the real handler. On success, store the response (status code + body) and mark status=done. On failure, you can keep the in_progress row so the client can retry.
4Reap. Purge keys older than TTL (typically 24 hours). Emit Idempotent-Replayed: true on cached responses so clients know it was a replay.

Try it: Your First Idempotency-Covered Endpoint

Versatile uses SignalR for real-time AI generation. The hub methods (GenerateStream, TestConnection) don't need idempotency — they're HTTP-notification-like patterns. But imagine Versatile had a POST /api/credits/purchase endpoint:

  1. What would the idempotency key represent? (The user clicking "Buy 100 credits" once, retried if the response never arrives)
  2. What happens if the user clicks "Buy" twice in quick succession? (Two different keys → two separate charges, which might be the correct UX or might need client-side debouncing)
  3. What happens if the server crashes after charging the card but before writing the response? (The key stays in_progress. On restart, the client retries with the same key. The server finds the key in_progress, checks if the charge already exists, and either completes or rejects.)

Why This Is Hard: Race Conditions

The claim step (step 1) is the hardest part to get right. Here's why:

The Check-Then-Insert Race

// BAD: Two concurrent requests can pass this check
if (!db.HasKey(key)) {
  db.InsertKey(key, "in_progress");
  // Both requests reach here!
}
return ExecuteHandler();

Two concurrent requests both see no key, both insert, and both execute the handler. This is a double charge.

The Fix: One Atomic Statement

// GOOD: Only one request wins the insert
try {
  db.Execute("INSERT INTO idempotency_keys (key, status)
              VALUES (@key, 'in_progress')
              ON CONFLICT DO NOTHING");
  if (rowsAffected == 0) {
    return ReplayExistingResponse(key);
  }
} catch {
  // SQL constraint violation — another request already owns this key
  return RetryLater();
}
return ExecuteHandler();
Pattern

Pessimistic Concurrency

INSERT ... ON CONFLICT DO NOTHING (or INSERT ... WHERE NOT EXISTS with a unique constraint) is the standard pattern. The database handles the atomicity. Never rely on application-level locking — it's too expensive and too easy to get wrong.

For very hot keys (thousands of concurrent requests for the same key), consider SELECT ... FOR UPDATE or a Redis-based lock with TTL. But start with the database constraint — it handles 99% of cases.

Idempotency in Versatile's Context

Let's look at how idempotency thinking applies to what Versatile actually does:

AI Provider Calls

src/services/providers/openai.js calls OpenAI's chat completions API. OpenAI doesn't expose a standard idempotency-key header. This means:

  • If the OpenAI call succeeds on their end but the HTTP response is lost, Versatile has no way to replay the response — it must re-send the request, which creates a new completion.
  • This is a at-most-once guarantee for AI calls: the function will not create a duplicate, but it might fail silent on a successful generation.
  • If you wanted at-least-once, you'd need to store the AI response yourself (e.g., by its prompt hash) and serve it from cache on retry.

SignalR Hub Methods

The GenerationHub methods (GenerateStream, TestConnection) are notification-style — the client opens a connection, the hub streams results, and network interruption means the client reconnects and starts a new stream. Each stream is ephemeral. Idempotency doesn't apply here because there's no persistent state being mutated by the hub call itself.

Where It Would Apply: Credit Purchase

If Versatile ever adds credit-based monetization (buy credits for AI generations), the purchase endpoint would need idempotency keys. The pattern would be:

  • Key = user_{userId}_{guid} — generated when the user confirms the purchase
  • Scope: per-user, so one user's purchase doesn't collide with another's
  • Race condition: a double-click on "Buy" sends two POSTs. The SQL constraint deduplicates.
  • Recovery: if the server crashes after charging Stripe but before writing the response, the client retries with the same key on reconnect, and the server sees the charge was already applied.

Advanced: Multi-Step Workflows

Some operations span multiple steps. For example, "buy 100 credits" might:

  1. Charge the user's card (Stripe API)
  2. Add 100 credits to their account (database)
  3. Send a confirmation email (email service)
  4. Log the transaction (analytics)

If step 1 succeeds but step 2 fails, the user has been charged but hasn't received credits. This is a partial failure, and idempotency of the outer request alone isn't enough to handle it.

Recovery Point Tracking

The solution is to track which steps completed: 

idempotency_keys (
  key            TEXT PRIMARY KEY,
  status         TEXT DEFAULT 'in_progress',  -- in_progress | done
  recovery_point TEXT,                        -- 'charged' | 'credited'
  response_code  INTEGER,
  response_body  JSONB
)

On retry, the server sees recovery_point = 'charged' and skips the charge, proceeding directly to step 2. This is the saga pattern adapted for idempotency.

Saga vs. Idempotency: A saga handles multi-step failure by running compensating transactions (refund the card if credit allocation fails). Idempotency with recovery points prevents the saga from re-executing completed steps. They're complementary — idempotency is the per-step safety net, sagas are the cross-step orchestration.

What You Learned

ConceptKey Takeaway
Idempotency N × operation = same state as 1 × operation. Not about responses, about state.
HTTP built-ins GET/PUT/DELETE are idempotent by spec. POST is not. Use the right verb first.
Idempotency-Key header Client generates UUID once, sends on every POST, reuses on retry.
Atomic claim Use INSERT ... ON CONFLICT DO NOTHING — never check-then-insert.
Scope by endpoint Keys should be scoped to (method, path, account_id) to prevent cross-tenant collisions.
Multi-step recovery Track recovery_point for workflows; skip completed steps on retry.

Your Turn: Design an Idempotent Endpoint

Open backend/Versatile.Api/ and look at the controllers and hubs. Pick one POST endpoint (or imagine one, like a hypothetical DebitCredits hub method). Design its idempotency:

  1. What is the idempotency key? (User-generated? Server-generated? When is it created?)
  2. Where is the key stored? (SQL table? Redis? In-memory cache?)
  3. What is the claim step? (SQL INSERT ... ON CONFLICT or Redis SET NX?)
  4. What happens on a partial failure? (Recovery point? Rollback? Manual intervention?)
  5. What is the TTL? (24 hours is the Stripe standard — long enough for retries, short enough to not accumulate forever.)

Write your design as a comment in the file or as a quick architecture decision record. The next time someone adds a payment-like endpoint, you'll have a template.
