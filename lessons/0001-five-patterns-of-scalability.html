Five Patterns of Scalability

How Versatile's architecture keeps growing without breaking.

Lesson 1 · 15 min · Reference
Prerequisites: You should have ARCHITECTURE.md open in another tab. This lesson walks through the codebase — open the linked files as you go.

"Scalability" sounds like a cloud-infrastructure word — more servers, more databases, more load balancers. In a codebase, scalability means something different: can the architecture grow without collapsing under its own weight? Can you add features, contributors, and users without everything becoming tangled?

Versatile makes 5 key architectural decisions that let it scale. Each one is a trade-off you can see in the code right now. Let's walk through them.

1. Offline-First with IndexedDB

Architecture Core Pattern

The Insight

Data is written to the browser first. The backend is optional — a sync target, not a dependency. The app works fully with zero servers running.

This is the single most impactful scalability decision in Versatile. Here's why:

If the backend were required...With offline-first...
Every page load hits the serverEvery page load reads IndexedDB (instant)
Scaling means scaling the databaseScaling means the database scales to one user (you)
Network = app is unusableNetwork = can't sync, but app works fine
Adding a feature needs backend + frontend changesAdding a feature can be entirely frontend

Where to see it

Open src/services/db-core.js. This is the IndexedDB schema — it defines 40+ tables for everything the app stores. Every Pinia store talks to this, not to an API.

Open src/services/sync-engine.js. This is the sync layer — it watches Dexie (IndexedDB) for changes and pushes them to the backend every 30 seconds. The backend is a mirror, not the source of truth.

Notice the syncStatus column on every table: pending-create, pending-update, synced. That's the state machine that makes offline-first work.

The scalability trade-off

You're trading simplicity (one database) for resilience (no dependency on backend). Conflict resolution becomes harder — if you edit on two devices, whose version wins? Last-write-wins (the current strategy). That works for a single-author writing tool but would need upgrading for multi-user collaboration.

Try it

Open DevTools → Application → IndexedDB → versatile. Browse the tables. Find a syncStatus field. Look at db-sections.js and db-scenes.js to see how the same data shape appears in both the Dexie table and the backend API.

2. Client-Side AI with Provider Abstraction

Architecture Pattern

The Insight

The browser calls AI providers directly. The server never becomes a bottleneck for the app's primary feature.

This is unusual. Most apps route AI calls through their backend (to control costs, log usage, etc.). Versatile does the opposite — all 5 AI providers (Ollama, OpenAI, Anthropic, Gemini, Groq) are called straight from the browser.

This means the backend doesn't need to scale with AI usage. If every user generates 100 chapters an hour, the backend is untouched — the load falls on the AI providers and the user's machine.

Where to see it

Open src/services/aiService.js. This is the single entry point. It exports four functions: aiGenerate, aiStream, aiGenerateStructured, aiTestConnection. Every feature in the app calls one of these.

Open src/services/providers/. Each file (openai.js, anthropic.js, ollama.js, etc.) exports the same four functions. The aiService.js dispatches to the right provider via a PROVIDER_MAP — adding a 6th provider means adding one file and one entry in the map.

Now look at the concurrency control in aiService.js around line 100:

// Counting semaphore per provider
const semaphores = {
  ollama: new Semaphore(1),  // local model — one at a time
  openai: new Semaphore(4),  // cloud — parallel is fine
  anthropic: new Semaphore(4),
  gemini: new Semaphore(4),
  groq: new Semaphore(4),
}

This sits at the single funnel point below all fan-outs. The semaphore is acquired per attempt inside the retry loop, not around it — so backoff doesn't hold the slot. This is the correct place for concurrency control: at the transport layer, not the task layer.

The scalability trade-off

You're trading security (API keys in the browser) for scalability (server never bottlenecks on AI). API keys are encrypted in localStorage with AES-GCM, but they're decrypted in JavaScript memory. Acceptable for a writing tool; you'd route through a server for a enterprise product.

Try it

Open src/services/providers/openai.js and compare it to src/services/providers/anthropic.js. Notice how the function signatures are identical. Find the PROVIDER_MAP in aiService.js — this is where a new provider would be registered.

3. Clean Architecture (Backend)

Architecture Pattern

The Insight

The backend is organized in concentric rings. Each layer knows about the layers inside it, never the ones outside. Dependencies point inward.

Open backend/ — the solution has four projects:

Versatile.Domain/         # 1. Entities + interfaces (pure POCOs, no dependencies)
Versatile.Application/    # 2. CQRS commands/queries, handlers, validators, DTOs
Versatile.Infrastructure/ # 3. EF Core, repositories, services, middleware
Versatile.Api/            # 4. Controllers, SignalR hubs, Program.cs

The rule: Domain knows nothing. API knows everything. Controllers can reference any layer. Domain code references nothing outside itself. This means:

  • You can swap the database (Infrastructure) without touching business logic (Application)
  • You can add a GraphQL API without touching the domain
  • You can test business logic without a database — just instantiate the handler and pass a mock repository

Where to see it

Open backend/Versatile.Api/Controllers/StoryController.cs. Notice it doesn't use a repository directly — it dispatches through IMediator:

[HttpPost]
public async Task<ActionResult<StoryDto>> Create(CreateStoryCommand command)
{
    var result = await _mediator.Send(command);
    return CreatedAtAction(nameof(GetById), new { id = result.Id }, result);
}

The CreateStoryCommand lives in Versatile.Application/. The handler (in Application) does the work. The controller just routes HTTP.

This is CQRS (Command Query Responsibility Segregation) — writes and reads use different paths. Currently applied to Story and Auth only; the remaining 36+ controllers inject services directly. An ongoing migration.

The scalability trade-off

You're trading boilerplate (4 projects, command classes, handler classes, DTOs) for decoupling (any layer can be replaced independently). This pays off when the team grows — new devs can add a feature by copying the pattern without understanding the whole system.

Try it

Open backend/Versatile.Application/Stories/Commands/CreateStory/. You'll find CreateStoryCommand.cs, CreateStoryHandler.cs, and CreateStoryValidator.cs. That's the CQRS triad — command, handler, validator. Trace one feature end-to-end: command → handler → repository → controller.

4. Modular Frontend via Features + Composables

Pattern

The Insight

Frontend code is grouped by feature, not by type. Components, composables, and stores for "story bible" live together. A new contributor can find everything for a feature in one place.

Open src/components/ — 20+ feature directories:

auth/   eval/   flow/   manuscript/   polish/
research/   revise/   shared/   spark/   story/
storybible/   storyshape/   ui/   voice-lab/   volume/

Each directory owns everything for that feature. The shared/ directory contains reusable primitives — Modal, ErrorBoundary, BaseButton — that any feature can import.

Now open src/composables/ — 65+ composables. These are Vue 3's Composition API functions that encapsulate reusable stateful logic. The extraction pattern (documented in ARCHITECTURE.md):

"Extract only when the service method has a clean 1:1 mapping with the inline block."

This means composables aren't extracted pre-emptively — they're carved out when a clear boundary emerges. This avoids the "pre-mature abstraction" trap while still converging toward a clean architecture.

Where to see it

Open src/stores/ — 15 Pinia stores. Each wraps one or more src/services/db-*.js modules. The store manages state, the db service manages persistence, and the components just render.

Try it

Pick one feature — say, "story bible". Find all the files that contribute to it: src/components/storybible/, the relevant store, the db service. Notice how each layer has a distinct job.

5. SignalR Hubs as Architectural Runway

Architecture Future-Proofing

The Insight

Real-time infrastructure is deployed but not yet used. The SignalR hubs for collaboration and server-side AI are wired up and authenticated — they're waiting for the frontend client that doesn't exist yet.

This is the opposite of YAGNI ("You Ain't Gonna Need It"). It's deliberate architectural runway:

  • /hubs/collaboration — supports cursor sync and content change broadcast for multi-user editing
  • /hubs/generation — supports server-side AI streaming for when client-side generation isn't enough

Both are JWT-authenticated. Both use SignalR groups (JoinStoryGroup/LeaveStoryGroup) so messages reach only the right users.

Where to see it

Open backend/Versatile.Api/Hubs/CollaborationHub.cs and backend/Versatile.Api/Hubs/GenerationHub.cs. The methods are defined, the streaming works (await foreach), but no frontend calls them yet.

Try it

Search for "GenerationHub" across the codebase. You'll find it referenced in the backend — but no @microsoft/signalr package in package.json. That's the gap: the runway is built, the plane hasn't landed yet.

Putting It Together

Each pattern buys you a different kind of scalability:

PatternScales...Trade-off
Offline-First Zero-infrastructure usage Conflict resolution complexity
Client-Side AI Server-free AI generation API keys in the browser
Clean Architecture Team + feature count Boilerplate and indirection
Feature Modules Codebase navigability Cross-feature coupling risks
SignalR Runway Future real-time features Dead code until the client arrives
