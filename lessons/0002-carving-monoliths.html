Carving Monoliths

Safe decomposition patterns for the generation pipeline.

Lesson 2 · 20 min · Requires: Lesson 1 · Reference
Why this matters: You're onboarding as a contributor. The single biggest barrier to adding features is the 1767-line composable that powers AI story generation. This lesson teaches you how to safely carve it up.

The Problem: One File to Rule Them All

Open src/composables/useVolumeStoryGenerator.js and scroll to the bottom. Count the exports: 

export function useVolumeStoryGenerator() {
  // ... 1700 lines ...
  return {
    startGeneration, resumeGeneration, stop, reset,
    confirmPlan, repairFailedScenes, completeGeneration,
    writeNextBatch, generateAnchor, generateMiddleScene,
    writeSceneWithGate, runParallelGeneration,
    regenerateScene, approveScene, rejectScene, rerequestScene,
    confirmSync,
    isReady, isGenerating, isPaused, phase, progress,
    writerParams, plan, events, ...
  }
}

Eighteen+ functions. Seven imported Pinia stores. Direct calls to IndexedDB, the AI service, the evaluator, and the quality scorer — all in one file. This is the cognitive load ceiling for the generation feature.

This matters for developer scalability:

  • Merge conflicts: Every PR that touches generation hits this file. Two devs can't work on separate scenes simultaneously.
  • Fear of change: Nobody wants to refactor a 1767-line function. Bugs hide in the noise.
  • Onboarding tax: New contributors must read the entire thing to understand any part of it.
  • Testing overhead: Unit tests for this composable need to mock 7 stores + IndexedDB + AI providers — so nobody writes them.
Anti-Pattern

The "Just Add One More Function" Trap

Every generation feature starts as "just one more async function" in this file. Individually, each addition is rational. Collectively, they create a file that no single person fully understands. This is how monoliths are born — not by design, but by accretion.

Try it

Open src/composables/useVolumeStoryGenerator.js. Count how many import statements reference Pinia stores (use*Store). Each one is a dependency that makes extraction harder. Note which functions use which stores.

The Seams: Where to Cut

A seam is a place where you can change behavior without changing the file itself. In the generation pipeline, seams are everywhere — you just need to see them.

Seam 1: The 6-Stage DAG

The generation pipeline already has a documented structure:

Bible → Network → Structure → Spine → Prose → Consistency

Each stage is a logical unit. Each has its own input, output, and success/failure criteria. But in the current code, all six stages live in one file, with the state that connects them scattered across refs.

The cut: Each stage could be a separate service class. The composable calls them in order, passing only what each stage needs.

Seam 2: The Export List

The return value of useVolumeStoryGenerator is a map of every capability the composable exposes. Every exported function is a candidate for extraction. Sort them by how many stores they touch:

  • stop / reset — touch only internal state (easy extraction)
  • confirmPlan — touches 2 stores + IndexedDB (medium)
  • generateAnchor — touches AI service + evaluator + IndexedDB (hard)
  • startGeneration — touches 7 stores + everything else (hardest)

Seam 3: Cross-Cutting Concerns

Some functions don't belong to any stage but are spread through the whole file:

  • Abort/stop logic (runSignal, isAborted, throwIfAborted) — 5 small functions that could live in one utility
  • Quality scoring (attemptScore, isCleanPass) — used by multiple stages, belongs in a scorer service
  • Rejection logging (logRejectedPattern) — an event, not core logic

Try it

Open the file again. Find the runSignal, isAborted, throwIfAborted, and isAbortError functions (lines ~123–147). They're pure, context-independent functions. If you extracted them to src/composables/generation/lifecycle/abort.js, how many lines would the main file lose? (Answer: ~25 lines immediately, plus every throwIfAborted() call becomes a clean import.)

The Pattern: Service Extraction

Versatile has already defined this pattern. It's documented in ARCHITECTURE.md under "Service Extraction." Here's the recipe:

Pattern

Service Extraction Recipe

  1. Identify a function with a clean boundary — it takes inputs, returns outputs, doesn't reach into the composable's internal refs
  2. Create a plain JS class in src/composables/generation/{domain}/
  3. The class receives only what it needs (refs as constructor args, not the entire store)
  4. The composable creates and calls the service — it becomes a thin coordinator
  5. Test separately: the service is a plain class, no Vue reactivity to mock

Notice what it does not say: extract everything. The rule is stricter than that:

"Extract only when the service method has a clean 1:1 mapping with the inline block."

This means: don't invent abstractions. Carve existing ones. If a block of code doesn't have a natural boundary, leave it in place until one emerges.

The Hardest Cut: Two-Phase Prose

The most complex part of the generation pipeline is the two-phase prose architecture:

  1. Phase 1 (Anchors): Generate "anchor" sentences for every scene in parallel — the first sentence of each narrative chunk
  2. Phase 2 (Middles): Fill in the middle of each scene, also in parallel, using the anchor as context

This lives in runParallelGeneration (line 832), generateAnchor (line 853), generateMiddleScene (line 989), and writeNextBatch (line 1126).

The challenge with extracting these:

  • writeNextBatch interleaves parallel AI calls with IndexedDB writes — each batch writes progress as it goes
  • generateAnchor references writerParams (a reactive ref from the composable) and manuscriptStore
  • The quality eval (attemptScore) is called inside writeSceneWithGate, which wraps both phases
Strategy

The Batched Extraction Approach

Don't try to extract the entire prose pipeline at once. Instead:

  1. First extract attemptScore / isCleanPass and logRejectedPattern — these are pure functions with no store dependencies
  2. Then extract generateAnchor — it's self-contained once you pass writerParams as an argument
  3. Then extract generateMiddleScene — same pattern
  4. Finally, runParallelGeneration becomes a thin coordinator that calls the extracted functions

Each step reduces the file by 100–200 lines and is independently testable.

Try it: Map the Data Flow

Open the file around line 832 (runParallelGeneration). Trace one scene generation end-to-end:

  1. runParallelGeneration calls writeSceneWithGate
  2. writeSceneWithGate calls generateAnchor or generateMiddleScene
  3. These call the AI service to generate text
  4. Then the quality evaluator scores it
  5. If it passes, it's written to IndexedDB via the manuscript store

Draw this flow on paper or in your head. Each arrow is a potential extraction boundary — if you can name what crosses that arrow, you can extract it.

Putting It Together: The Decomposition Roadmap

Here's what a fully decomposed generation pipeline looks like. This is a destination, not something to build in one sitting:

CurrentTargetLines Saved
runSignal, isAborted, throwIfAborted, isAbortError generation/lifecycle/abort.js ~25
attemptScore, isCleanPass generation/pipeline/scorer.js ~20
logRejectedPattern generation/writing/rejection-logger.js ~15
generateAnchor generation/writing/anchor-generator.js ~135
generateMiddleScene generation/writing/middle-generator.js ~140
writeSceneWithGate generation/writing/gated-writer.js ~120
writeNextBatch generation/writing/batch-writer.js ~215
runParallelGeneration generation/pipeline/parallel-runner.js ~125
confirmPlan generation/pipeline/plan-confirmer.js ~175
repairFailedScenes generation/pipeline/scene-repairer.js ~75
completeGeneration generation/pipeline/generation-finalizer.js ~50
Remaining coordinator useVolumeStoryGenerator.js ~1100 lines → ~400 lines

The composable shrinks from 1767 lines to ~400. It stops doing work and starts coordinating work — calling services, handling the orchestration, keeping the reactive state. Each service is independently testable, independently changeable, and independently understandable.

The metric to watch: Not lines of code, but imports per file. A well-decomposed file imports 3–5 things. The current composable imports 15+. Each extraction should reduce that number.

What You Learned

ConceptKey Takeaway
Developer scalability It's about cognitive load, not server throughput. A 1767-line file is the bottleneck.
Seams Natural boundaries exist (6-stage DAG, export list, cross-cutting concerns). Find them, don't invent them.
Service extraction Extract only when there's a 1:1 mapping. Services are plain JS classes, composables are thin coordinators.
Batched decomposition Start with pure functions (low risk), then move to self-contained operations, then the coordinator.
Testing wins Extracted services are plain classes — no Vue reactivity, no Pinia mocks needed.
