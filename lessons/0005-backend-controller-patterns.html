Backend Controller Patterns

How Versatile's API layer works — and the 5 inconsistencies we found.

Lesson 5 · 20 min · MISSION.md
Prerequisites: Lesson 4 (Multi-Tenancy Fundamentals). You should be comfortable opening files in backend/Versatile.Api/Controllers/.

We just analyzed every backend controller in Versatile. We found 5 controllers, all following MediatR CQRS — plus 5 inconsistencies that need attention before multi-tenancy lands.

This lesson teaches you what each layer in the backend does, how a request flows through them, and why the inconsistencies matter.

1. The Five Layers of Clean Architecture

Versatile's backend follows the Jason Taylor Clean Architecture template. Open backend/Versatile.sln in your head — these are the projects:

LayerProjectJob
Domain Versatile.Domain/ Entities, enums, value objects. Zero dependencies. The "what."
Application Versatile.Application/ CQRS commands, queries, handlers, validation. The "orchestration." Depends only on Domain.
Infrastructure Versatile.Infrastructure/ EF Core DbContext, repositories, JWT auth, email, AI services. The "how." Depends on Application.
API Versatile.Api/ Controllers, middleware, SignalR hubs, DI configuration. The "entry point." Depends on Infrastructure and Application.
Shared Versatile.Shared/ DTOs, constants, extension methods used across layers.

The Dependency Rule

Dependencies point inward. Domain knows nothing about Infrastructure. API knows about Infrastructure through interfaces defined in Application. This is what makes the architecture testable — you can swap PostgreSQL for SQLite by changing one project.

2. How a Request Flows Through

Let's trace a Chapter create from HTTP to database and back. Open the files as we go.

Step 1: Controller

Open backend/Versatile.Api/Controllers/ChapterController.cs:

[ApiController]
[Route("api/stories/{storyId}/chapters")]
[Authorize]
public class ChapterController : ControllerBase
{
    private readonly IMediator _mediator;

    public ChapterController(IMediator mediator)
    {
        _mediator = mediator;
    }

    [HttpPost]
    public async Task<IActionResult> Create(
        Guid storyId,
        [FromBody] CreateChapterCommand command)
    {
        command.StoryId = storyId;
        var result = await _mediator.Send(command);
        return Ok(result);
    }
}

The controller's job is minimal: extract HTTP data, send a command to MediatR, return the response. It doesn't validate, doesn't talk to the database, doesn't know about infrastructure.

Step 2: MediatR Pipeline

MediatR isn't just "send and forget." Versatile has a pipeline wrapped around every command:

// Pseudocode — what MediatR runs in order:
1. ValidationBehavior<TRequest, TResponse>  — FluentValidation
2. LoggingBehavior<TRequest, TResponse>     — Serilog
3. PerformanceBehavior<TRequest, TResponse> — timing & slow-call warnings
4. The handler itself                         — your business logic

This pipeline behavior pattern is why cross-cutting concerns (validation, logging, timing) don't pollute your handler code. Each behavior is a separate class, registered once in DI, applied to every command automatically.

Step 3: Handler

Open backend/Versatile.Application/Chapters/Commands/CreateChapter/CreateChapterHandler.cs:

public class CreateChapterHandler : IRequestHandler<CreateChapterCommand, ChapterDto>
{
    private readonly IOrganizationContext _context;
    private readonly IMapper _mapper;

    public async Task<ChapterDto> Handle(
        CreateChapterCommand request,
        CancellationToken cancellationToken)
    {
        var chapter = new Chapter
        {
            StoryId = request.StoryId,
            Title = request.Title,
            SortOrder = request.SortOrder
        };

        _context.Chapters.Add(chapter);
        await _context.SaveChangesAsync(cancellationToken);

        return _mapper.Map<ChapterDto>(chapter);
    }
}

The handler: creates a domain entity, saves it via DbContext, maps to a DTO. That's it. No attribute routing, no serialization concerns, no HTTP context.

Step 4: Infrastructure

The IOrganizationContext interface lives in Application (the contract) but is implemented by OrganizationContext in backend/Versatile.Infrastructure/Data/OrganizationContext.cs — that's the actual EF Core DbContext with all 36 DbSets.

Step 5: Back through the pipeline

The response travels back through the same pipeline behaviors in reverse order, then the controller wraps it in Ok(result) (HTTP 200).

Pattern

Why MediatR instead of calling a service directly?

Direct calls would still work. But MediatR gives you:

  • Pipeline behaviors — validation, logging, timing, and (soon) tenant-context injection all wrapped around every command without touching the handler
  • Single responsibility — a handler does one thing. A service class tends to grow
  • Testability — handlers are easy to unit-test. The pipeline is easy to integration-test

3. The 5 Issues We Found

During the analysis, we identified 5 things that need standardization before multi-tenancy. Let's walk through each one.

Issue 1: Dual Error Middleware

Versatile has two global error handlers:

  • backend/Versatile.Api/Middleware/ExceptionMiddleware.cs — classic RequestDelegate middleware. Catches exceptions in the pipeline.
  • backend/Versatile.Api/Middleware/ExceptionHandler.cs — minimal API pattern IExceptionHandler. Same job, different mechanism.

Both are registered in Program.cs. Which one runs first? Depends on the pipeline order. Which one handles which exception? Unclear. When you add tenant-context middleware later, you need to know exactly where errors get caught — otherwise a tenant resolution failure might get swallowed by the wrong handler.

Trace it: Open both files

  1. Open ExceptionMiddleware.cs and ExceptionHandler.cs
  2. Note the exception types each handles
  3. Open Program.cs and find both registrations
  4. Question: if a ValidationException is thrown in a handler, which catches it first?

Issue 2: Auth Inconsistency in SceneController

Most controllers are pure MediatR dispatchers. But open backend/Versatile.Api/Controllers/SceneController.cs — it also injects IOrganizationContext directly:

public SceneController(IMediator mediator, IOrganizationContext context)
{
    _mediator = mediator;
    _context = context;  // Direct DbContext in the API layer
}

This breaks the Clean Architecture dependency rule. The API layer should depend on Application interfaces, not Infrastructure directly. For multi-tenancy, this matters because tenant-context injection happens in the MediatR pipeline — bypassing MediatR means bypassing tenant filtering.

Issue 3: Split DI Registration

Dependencies are registered in two places:

  • Convention scanning: HandlersRegistration, ValidatorsRegistration, DbContextRegistration — these automatically find and register all MediatR handlers, FluentValidation validators, and EF configurations
  • Manual registrations: Everything in Program.cs after each services.Add*() call

This is fine structurally, but the IOrganizationContext singleton workaround in Program.cs is a red flag — it was needed because some code path needed the context outside a scoped request. For multi-tenancy, every resolution path must go through the scoped tenant context.

Issue 4: No Repository Abstraction

Handlers use IOrganizationContext (the EF DbContext) directly. No repository layer between them and the database.

This is a deliberate trade-off in the Jason Taylor template — for simple CRUD, a repository layer adds indirection without benefit. But for multi-tenancy, you need to consider: does every handler correctly filter by tenant, or does a repository guarantee it?

PostgreSQL Row-Level Security (RLS) can enforce this at the database level, making a repository layer less critical. But if you skip RLS, every handler is responsible for its own WHERE tenant_id = @current — and that means every handler is a potential data leak.

Issue 5: Missing OpenAPI Metadata

public async Task<IActionResult> Create(...)  // IActionResult — Swagger can't infer the shape

All controllers return IActionResult instead of ActionResult<T>. Swagger generates documentation from return types — IActionResult tells Swagger nothing. The fix is [ProducesResponseType] attributes or switching to ActionResult<ChapterDto>.

This matters for multi-tenancy because you'll want clear API documentation when third-party developers or frontend clients integrate with the org endpoints.

Putting It Together

The backend is well-structured — MediatR CQRS, pipeline behaviors, Clean Architecture layers. The 5 issues are accumulated debt from single-user development, not fundamental design flaws. Each one becomes a liability when scaling to multi-tenancy:

IssueMulti-Tenancy RiskFix Priority
Dual error middlewareTenant resolution errors get swallowed by the wrong handlerBefore Phase 2
Auth inconsistencySceneController bypasses MediatR pipeline (and future tenant context injection)Before Phase 1
Split DI registrationNon-scoped tenant context could leak across requestsDuring Phase 2
No repository abstractionEvery handler must remember to filter by tenant_idMitigate with RLS (Phase 2), not a repository
Missing OpenAPI metadataUnclear API contracts for org-scoped endpointsLow — cosmetic but useful

The advice

Fix Issue 2 (SceneController) before starting Phase 1. Move the direct IOrganizationContext lookup into a MediatR query handler. That way when tenant context gets injected into the pipeline, SceneController gets it automatically.

Fix Issue 1 (error middleware) before Phase 2. Pick one pattern (IExceptionHandler is the modern choice) and remove the other. Then tenant-context middleware errors flow through a single, predictable path.
